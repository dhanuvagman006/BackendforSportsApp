# SportyQo Backend

Production-grade REST + WebSocket API for **SportyQo** — the cricket-first sports
networking and player-performance app. Built to the contract in
[`docs/SportyQo_API_Documentation.md`](docs/SportyQo_API_Documentation.md) and
[`docs/SportyQo_Backend_Requirements_Summary.md`](docs/SportyQo_Backend_Requirements_Summary.md).

**Stack:** FastAPI · SQLAlchemy 2 (async) · PostgreSQL 16 · Alembic · PyJWT · argon2id

---

## Quick start

### Docker (one command)

```bash
docker compose up --build
# API      → http://localhost:8000
# Swagger  → http://localhost:8000/docs
```

### Local

```bash
# 1. Postgres 16 with a sportyqo user/db and the citext extension
createuser sportyqo -P            # password: sportyqo
createdb -O sportyqo sportyqo
psql -d sportyqo -c 'CREATE EXTENSION citext;'

# 2. Install & configure
pip install -r requirements.txt
cp .env.example .env

# 3. Migrate, seed, run
alembic upgrade head
python -m scripts.seed --demo     # optional demo coach/player/league
uvicorn app.main:app --reload
```

Demo logins after `--demo`: `coach@demo.sportyqo` / `player@demo.sportyqo`, password `Demo1234`.

### Tests

```bash
createdb -O sportyqo sportyqo_test
psql -d sportyqo_test -c 'CREATE EXTENSION citext;'
pytest            # 36 tests: auth, scoring, leagues, matches, social, media
```

## Core design decisions

### Qo Score — append-only ledger
`qo_score_events` is immutable; `qo_scores.score` is a **materialised SUM** that is
recomputed on every write, never mutated in place. There is *no* direct score-write
path anywhere — match submissions, recommendations, post rewards and even admin
corrections all append ledger events. Match submission requires an
`Idempotency-Key` header; replays return the stored result and never double-award.

Scoring weights live in config (`.env`), not code:

| Event | Points (default) |
|---|---|
| Match participation | 5 |
| Per run | 0.5 (floored) |
| Per wicket | 20 |
| Per catch | 10 |
| Man of the Match | +25 |
| Winning side | +15 |
| Dugout post | 12 |
| Coach recommendation | 25 (1 per coach-player / 30 days) |

Card tiers (8, Purple → Golden Pro) are **server-owned config** served from
`GET /v1/config/card-tiers` — thresholds can be tuned without an app release.

### Server-minted identifiers
- **Player IDs** `P{YY}{NNN}` — per-year sequence table under `SELECT … FOR UPDATE`;
  allocation is idempotent per user.
- **League codes** `FALC-16-24` style — unique index + collision retry, widening
  the space if retries are exhausted.

### Auth
Email+password (argon2id) with a **role-checked login** (`403 ROLE_MISMATCH`).
JWT access tokens (15 min) + **rotating refresh tokens** (30 days) stored hashed
in `sessions` with a family id — reusing a rotated token revokes the whole family
(`401 TOKEN_REUSED`). Coach onboarding uses SMS OTP (TTL 300 s, 5 attempts,
3 sends / 15 min per phone; the code is echoed as `dev_code` outside production).
Password reset + email verification are console-emailed in dev. Account deletion
is a soft delete with a 30-day grace window (DPDP).

### Privacy (minors)
`dob`, `school` and `phone` never appear in public payloads — only `age_group`
and city-level location. The `private_profile` setting hides profile tabs from
non-followers. Uploaded images are re-encoded to strip EXIF (incl. GPS).

### Providers
SMS, email, push and storage sit behind interfaces selected via env:
console/local-disk in development; MSG91/SMTP/FCM/S3 slot in for production
without touching call sites. Private files (certification docs) are served only
through HMAC-signed URLs.

### Realtime
`wss://…/v1/ws?token=<access_token>` with JSON frames
`{"channel","event","data"}`. Channels per spec §9 with per-channel
authorization (own notifications/score, league members, conversation
participants, media owner). The manager is in-process; back it with Redis
pub/sub when running multiple instances.

### Error envelope
Every error (including validation) returns spec §11's shape with a request id:

```json
{ "error": { "code": "ROLE_MISMATCH", "message": "…", "field": null, "request_id": "req_…" } }
```

## API surface (v1)

~60 endpoints matching the spec's endpoint contracts: auth (register/login/
refresh/logout/OTP/password/email-verify), profiles + sport selection,
settings, public profiles + follow, player dashboard / qo-score / performance /
matches / playbook, coach dashboard / roster / recommendations / certification /
playbook, leagues (create/join/exit/detail/code/players/matches), match points,
dugout feed + posts + likes/comments/bookmarks/share, friend requests, DMs,
media (direct + presigned + signed serving), notifications + devices, config,
and an `X-Admin-Key`-guarded admin surface (certification review, audit-logged
point corrections). Interactive docs at `/docs`.

## Operations

- **Migrations:** `alembic upgrade head` (the Docker image runs this on boot).
- **Rankings cron:** `python -m scripts.recompute_rankings` (e.g. every 30 min);
  rankings are also recomputed inline after score mutations.
- **Health:** `GET /health`.

## Deferred for production hardening

Wired-behind-interface but not activated: real MSG91/Twilio, SES/SMTP, FCM/APNs
fan-out, S3 + CloudFront (presigned flow currently targets a local PUT
endpoint), HLS transcode (dev marks video `ready` immediately), Redis pub/sub
for multi-instance WebSockets, purge job for soft-deleted accounts past the
grace window, weekly coach performance-report + match-reminder crons, and Sentry.
